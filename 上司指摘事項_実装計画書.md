# 上司指摘事項の実装計画

## Context
上司にアプリを確認してもらい、10件の指摘事項を受けた。OCRバグ修正、ログイン機能、名刺共有、会社情報リンク改善など、大規模な機能追加が必要。Sansanの名刺管理を参考に設計。

## 指摘事項一覧
1. NEOJAPAN名刺画像でOCR解析が失敗する
2. ログイン機能（アプリ起動時にログイン画面表示）
3. ログインまたはユーザー新規作成を選択可能に
4. ID（メール）とパスワードで管理
5. ユーザーごとのデータ分離
6. 他のユーザーに名刺データを共有できる
7. 同じ名刺を他のユーザーも持っている場合、それが見てわかるようにする
8. 会社名から会社HPを探して会社名にリンク紐づけ
9. 「Webサイト」→「関連サイト」に名称変更
10. 名刺記載URLが会社サイトなら会社名リンクに、製品サイトなら関連サイトとして登録
11. 一覧の検索で音声入力に対応してほしい

---

## フェーズ1: OCRバグ修正 + ラベル名称変更 + 音声検索

### 1-A. OCRバグ修正（指摘#1）

**`src/lib/ocr.ts`**
- 正規表現 `/^data:(image\/\w+);base64,(.+)$/` を `/^data:(image\/[\w+.\-]+);base64,(.+)$/s` に改善

**`src/app/api/ocr/route.ts`**
- JSON抽出にフォールバック追加: コードブロックマッチ失敗時に `/\{[\s\S]*\}/` で直接抽出
- JSON.parse失敗時のエラーログ詳細化（レスポンス先頭200文字をログ出力）

### 1-B. ラベル名称変更（指摘#9）

| ファイル | 変更 |
|---|---|
| `src/components/card-detail.tsx` | `label="Webサイト"` → `label="関連サイト"` |
| `src/components/business-card-form.tsx` | `label: "Webサイト"` → `label: "関連サイト"` |
| `src/app/api/ocr/route.ts` | プロンプト内の説明を更新 |

### 1-C. 検索の音声入力対応（指摘#11）

**`src/types/speech-recognition.d.ts`（新規）**
- Web Speech API (`SpeechRecognition`, `webkitSpeechRecognition`) のTypeScript型定義

**`src/components/search-bar.tsx`**
- マイクボタンを検索バー右端に追加（Lucide React `Mic` / `MicOff` アイコン）
- Web Speech API (`SpeechRecognition`) で音声認識を実装
  - `recognition.lang = "ja-JP"`（日本語）
  - `recognition.continuous = false`（1回の発話で終了）
  - `recognition.interimResults = true`（途中結果も表示）
- マイクボタン押下で録音開始、再度押下で停止
- 録音中はマイクアイコンをアニメーション（赤い点滅等）
- 音声認識結果を `onChange` に渡して検索を発火
- 非対応ブラウザではマイクボタンを非表示
- エラー時はトーストで通知

## フェーズ3: 会社情報・リンク改善（指摘#8, #10）

### 3-A. データモデル拡張
**`src/types/business-card.ts`**
- `companyUrl: string` フィールドを追加（会社HPのURL）

### 3-B. 会社HP自動取得（Google Custom Search API使用）
- Google Cloud で Custom Search API を有効化、APIキーを取得
- `.env.local` に `GOOGLE_CSE_API_KEY` と `GOOGLE_CSE_ID` を追加
- **`src/app/api/company-url/route.ts`（新規）** — 会社名から会社HP URLを検索するAPIルート
  - `GET /api/company-url?company=株式会社ネオジャパン`
  - Google Custom Search APIで「{会社名} 公式サイト」を検索、1件目のURLを返す
  - 無料枠: 1日100クエリ、超過は$5/1000クエリ

### 3-C. OCRプロンプト修正
**`src/app/api/ocr/route.ts`**
- プロンプトのJSONスキーマに `websiteType` を追加
  - `"websiteType": "名刺記載URLの種別（'corporate' or 'product'）"`
- OCR結果の `websiteType` に基づいてURL振り分け:
  - `corporate` → `companyUrl` に設定
  - `product` / その他 → `website`（関連サイト）に設定

### 3-D. OCR後に会社HP自動検索
**`src/lib/ocr.ts`**
- OCR解析完了後、`companyUrl` が空で `company` がある場合、`/api/company-url` を呼んで会社HPを自動取得

### 3-E. UI修正
**`src/components/card-detail.tsx`**
- 会社名の `InfoRow` に `href={card.companyUrl}` を追加（会社名クリックで会社HPに遷移）

**`src/components/business-card-form.tsx`**
- `fields` 配列に `companyUrl` フィールドを追加（会社名の直後）

## フェーズ4: 名刺共有機能（指摘#6, #7）

### 4-A. データモデル（Sansan参考: 一元管理 + sharedWith[]）
**`src/types/business-card.ts`** に追加:
- `ownerId: string` — 名刺の所有者UID
- `sharedWith: string[]` — 共有先のUID配列

### 4-B. インデックスコレクション（重複検出用）
名刺保存時に `cardIndex/{autoId}` にインデックスを作成:
```
{
  email: "saito@neo.co.jp",
  lastName: "齋藤",
  firstName: "晴義",
  company: "株式会社ネオジャパン",
  ownerId: "uid_xxx",
  cardId: "card_yyy",
  ownerEmail: "user@example.com"
}
```

### 4-C. storage.ts に追加する関数
- `shareCard(userId, cardId, targetEmail)` — メールからユーザー検索 → sharedWith に追加
- `getSharedCards(userId)` — 自分に共有された名刺を取得
- `findDuplicateOwners(userId, card)` — 同じ名刺を持つ他ユーザーを検索（メール優先、なければ氏名+会社名）
- `unshareCard(userId, cardId, targetUserId)` — 共有解除

### 4-D. UI
- **`src/components/share-dialog.tsx`（新規）** — メールアドレス入力 → 共有実行のダイアログ
- **`src/app/cards/[id]/page.tsx`** — 「共有」ボタン追加
- **`src/app/page.tsx`** — 「共有された名刺」セクション追加
- **`src/components/card-detail.tsx`** — 「この名刺を持っているユーザー」セクション追加

### 4-E. Firestoreセキュリティルール
```
match /users/{userId}/businessCards/{cardId} {
  allow read, write: if request.auth != null && request.auth.uid == userId;
  allow read: if request.auth != null && request.auth.uid in resource.data.sharedWith;
}
match /cardIndex/{indexId} {
  allow read: if request.auth != null;
  allow write: if request.auth != null && request.auth.uid == request.resource.data.ownerId;
}
match /users/{userId} {
  allow read: if request.auth != null;
  allow write: if request.auth != null && request.auth.uid == userId;
}
```

## 実装順序

```
フェーズ1（即時対応）    → OCRバグ修正 + ラベル変更 + 音声検索
フェーズ2（認証基盤）    → ログイン・ユーザー管理・データ分離
フェーズ3（会社情報改善） → companyUrl追加、Google検索API連携、URL振り分け
フェーズ4（共有機能）    → 名刺共有、重複検出・表示
```

## 検証方法
1. NEOJAPAN名刺含む複数画像でOCR成功を確認
2. サインアップ → ログイン → 名刺CRUD → ログアウト → リダイレクト確認
3. 異なるアカウントでデータが分離されていることを確認
4. OCR結果に会社HPが含まれ、会社名がリンク化されていることを確認
5. 名刺共有 → 共有先で閲覧可能（読み取り専用）を確認
6. 同じメールアドレスの名刺を持つユーザーが表示されることを確認
7. 検索バーのマイクボタンで音声入力→検索が動作すること（Chrome/Edge）
8. `npm run build` でエラーなし
