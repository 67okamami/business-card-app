# Firebase Authentication 導入計画

## Context
現在のFirestoreセキュリティルールは `allow read, write: if true`（全開放）で、誰でもデータの読み書きが可能な状態。メール/パスワード認証を導入し、ログインユーザーのみが自分のデータにアクセスできるようにする。

---

## 競合調査: Sansan / Eight のログイン設計（2026-03-16調査）

### Sansan（法人向け）
- ログイン方法: メール+パスワード、SAML SSO（Microsoft Entra ID等）、2FA（TOTP or メールOTP）
- 新規登録: **管理者がユーザーを追加**するエンタープライズ型（セルフサインアップなし）
- ログアウト: Webは右上アカウントメニュー、モバイルは設定メニュー内
- セッション管理: 管理者が自動ログアウト期間・アクセスデバイスを制御可能

### Eight（個人向け）
- ログイン方法: メール+パスワード、Apple ID（Touch ID/Face ID）、Facebook
- 新規登録: セルフ登録（メール認証 → 名刺プロフィール登録 → パスワード設定）
- ログアウト: Web設定画面から。複数デバイスの遠隔ログアウトも可能
- セッション管理: アプリ再起動後も自動ログイン維持

### 今回のアプリの方針
個人・小規模向けのため **Eight に近いフロー** を採用する。

| 項目 | 採用方針 |
|---|---|
| 登録方法 | メール + パスワードによるセルフ登録 |
| ログイン | メール + パスワード |
| 自動ログイン | Firebase Auth のセッション永続化で対応 |
| ログアウト | ヘッダー右上のアカウントメニューから |

---

## 方針

- Firebase Authentication（メール/パスワード）でログイン機能を追加
- Firestoreのデータ構造を `users/{userId}/businessCards/{cardId}` のサブコレクションに変更し、ユーザーごとにデータを分離
- セキュリティルールを `request.auth.uid == userId` に強化
- 未ログイン時はログインページにリダイレクト

## 修正・新規ファイル

### 1. Firebase Authenticationの有効化（Firebase MCP経由）
- `firebase_init` でメール/パスワード認証プロバイダーを有効化

### 2. `src/lib/firebase.ts` — Auth追加
- `getAuth()` をインポート・エクスポート

### 3. `src/lib/auth.ts` — **新規** 認証関数
- `signUp(email, password)` — ユーザー登録
- `signIn(email, password)` — ログイン
- `signOutUser()` — ログアウト
- `getCurrentUser()` — 現在のユーザー取得

### 4. `src/contexts/auth-context.tsx` — **新規** 認証コンテキスト
- `AuthProvider` — `onAuthStateChanged` で認証状態を監視
- `useAuth()` フック — `{ user, loading }` を返す

### 5. `src/app/layout.tsx` — AuthProvider適用
- クライアントコンポーネント化して `AuthProvider` でラップ
- metadataはServer Componentのままにするため、レイアウトを分割

### 6. `src/app/auth/login/page.tsx` — **新規** ログインページ
- メールアドレス・パスワード入力フォーム
- サインアップページへのリンク
- 既存UIコンポーネント（Button, Input）を活用

### 7. `src/app/auth/signup/page.tsx` — **新規** サインアップページ
- メールアドレス・パスワード・パスワード確認入力
- ログインページへのリンク

### 8. `src/lib/storage.ts` — ユーザーID対応
- コレクションパスを `businessCards` → `users/{userId}/businessCards` に変更
- 全関数に `userId` パラメータを追加
- 呼び出し元で `useAuth().user.uid` から取得

### 9. `src/app/page.tsx` 等の各ページ — 認証チェック追加
- `useAuth()` で認証状態を確認
- 未認証時はログインページにリダイレクト
- ヘッダーにログアウトボタンを追加

### 10. `firestore.rules` — セキュリティルール強化
```
match /users/{userId}/businessCards/{cardId} {
  allow read, write: if request.auth != null && request.auth.uid == userId;
}
```

## 実装の流れ

1. Firebase MCP経由でメール/パスワード認証を有効化
2. `src/lib/firebase.ts` にAuth追加
3. `src/lib/auth.ts` 作成
4. `src/contexts/auth-context.tsx` 作成
5. `src/app/layout.tsx` にAuthProvider適用
6. ログイン・サインアップページ作成
7. `src/lib/storage.ts` をサブコレクション対応に修正
8. 各ページに認証チェック・リダイレクト追加
9. ヘッダーにログアウトボタン追加
10. `firestore.rules` 更新・デプロイ
11. 動作確認・Vercel再デプロイ

## 注意事項

- 既存のFirestoreデータ（`businessCards/`直下）は新構造（`users/{uid}/businessCards/`）に移行されないため、ログイン後は空の状態からスタートになる
- Firebase Authenticationは無料のSparkプランで利用可能（月5万ユーザーまで無料）

## 検証方法

1. サインアップ画面でアカウント作成
2. ログイン画面でログインし、一覧ページに遷移すること
3. 名刺の登録・閲覧・編集・削除が正常に動作すること
4. ログアウト後、一覧ページにアクセスするとログインページにリダイレクトされること
5. Firestoreコンソールでデータが `users/{uid}/businessCards/` に保存されていること
6. `npm run build` でビルドエラーがないこと
