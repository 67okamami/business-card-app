# 名刺管理Webアプリ 実装計画書

## 概要

手入力とOCR（Tesseract.js）による名刺登録に対応した、モバイルファースト＋PC対応の名刺管理Webアプリケーション。
モバイルでの操作性を最優先しつつ、PCブラウザでも快適に利用できるレスポンシブデザインを採用する。

## 実装ステップ

### Step 1: プロジェクト初期化

- Next.js プロジェクトを作成（TypeScript, Tailwind CSS, App Router）
- shadcn/ui を初期化し、必要なコンポーネントをインストール
  - Button, Input, Label, Card, Dialog, Textarea, Badge, Separator
- Tesseract.js, Lucide React をインストール

### Step 2: 型定義・データ層の実装

- `types/business-card.ts` — `BusinessCard` インターフェース定義
- `lib/storage.ts` — localStorage を使ったCRUD関数群
  - `getCards()` / `getCardById(id)` / `saveCard(card)` / `updateCard(card)` / `deleteCard(id)` / `searchCards(query)`

### Step 3: 共通フォームコンポーネントの作成

- `components/business-card-form.tsx`
  - 初期値（空 or OCR結果）を受け取り、バリデーション付きで編集・保存
  - モバイルファースト：大きめのタッチターゲット、セクション分けしたレイアウト
  - PC版：2カラムグリッドで効率的な入力レイアウト（姓名、電話番号等を横並び）
  - 名刺画像のプレビュー表示対応

### Step 4: OCR機能の実装

- `components/ocr-capture.tsx`
  - カメラ起動（`<input type="file" accept="image/*" capture="environment">`）またはファイル選択
  - 撮影/選択した画像のプレビュー表示
  - 「解析する」ボタンで OCR 実行
- `lib/ocr.ts`
  - Tesseract.js による画像解析（日本語 + 英語対応: `jpn+eng`）
  - 解析進捗のコールバック（ローディング表示用）
- `lib/ocr-parser.ts`
  - OCR生テキストからメール・電話・会社名等をパターンマッチで抽出
  - 正規表現: メール(`/@/`), 電話(`/0\d{1,4}-?\d{1,4}-?\d{3,4}/`), URL(`/https?:\/\//`) 等

### Step 5: ページ実装

- **一覧ページ** (`app/page.tsx`)
  - 検索バー + 名刺カードのリスト表示
  - モバイル: 1カラムリスト + フローティング「新規登録」ボタン（FAB）
  - PC: 3カラムグリッド + ヘッダー右上に「新規登録」ボタン
  - 登録方法選択ダイアログ（手入力 / 写真から）
- **新規登録ページ** (`app/cards/new/page.tsx`)
  - クエリパラメータ `?method=manual|ocr` で分岐
  - OCR: まず `ocr-capture` → 解析結果を `business-card-form` に渡す
  - 手入力: 空の `business-card-form` を表示
- **詳細ページ** (`app/cards/[id]/page.tsx`)
  - モバイル: 1カラム縦積みレイアウト
  - PC: 2カラム（左: 名刺画像40%、右: 詳細情報60%）
  - 名刺情報の表示、電話・メールはタップ/クリックで発信/送信可能
  - 編集・削除ボタン
- **編集ページ** (`app/cards/[id]/edit/page.tsx`)
  - 既存データを `business-card-form` に流し込んで編集

### Step 6: UI仕上げ・レスポンシブ対応

- レスポンシブ対応（モバイルファースト + PC対応）
  - ブレークポイント: `md`(768px) で モバイル/PC レイアウトを切り替え
  - モバイル: 1カラム、FAB、大きめタッチターゲット
  - PC: グリッドレイアウト、ヘッダーボタン、2カラムフォーム、最大幅制限（1200px）
- ローディング・空状態・エラー状態のUI
- 削除時の確認ダイアログ
- トースト通知（保存成功/削除完了）

## OCR解析フロー

```
画像取込 → プレビュー表示 → 「解析する」押下
  → Tesseract.js 実行（プログレスバー表示）
  → 生テキスト取得
  → ocr-parser でフィールド抽出
  → フォームに自動入力
  → ユーザーが修正 → 「保存」
```

## OCR技術スタックに関する検討メモ

### 現行方式: Tesseract.js + 正規表現パーサー
- **OCR**: Tesseract.js v7（ブラウザ上で動作、`jpn+eng`）
- **フィールド抽出**: `ocr-parser.ts` で各行をパターン分類し、正規表現でフィールドに割り当て
- **メリット**: APIキー不要、オフライン動作可能、無料
- **制限**: 正規表現ベースのため、会社名に地名が含まれる場合（例: 「北海道フーズ株式会社」の「道」が住所と誤判定される）等のエッジケースがある

### 検討した代替案: Claude Vision API（2026-03-12検討）
- 名刺画像をClaude Vision API（claude-sonnet-4-6）に直接送信し、1回のAPI呼び出しでセマンティックにフィールド抽出する方式
- **メリット**: 文脈を理解した高精度なフィールド分類が可能。正規表現のエッジケースが根本的に解消される
- **デメリット**: Anthropic APIキーが必要、API呼び出し費用が発生（1枚あたり約0.7円）、ネットワーク接続必須
- **見送り理由**: 現行方式で実用上十分と判断。将来的に精度が問題になった場合の有力な改善候補として記録しておく

## 画像保存方式に関する検討メモ（2026-03-13検討）

### 現行方式: data URLをFirestoreに直接保存
- 名刺画像をBase64エンコードしたdata URL文字列として、Firestoreドキュメントの`imageUrl`フィールドに保存
- **メリット**: 追加サービス不要、Sparkプラン（無料）で運用可能、実装がシンプル
- **制限**: Firestoreの1ドキュメント上限1MBのため高解像度画像は保存不可。一覧取得時に画像データも毎回読み込まれるため通信量・課金が増加する可能性あり

### 検討した代替案: Firebase Storage
- 画像をFirebase Storageにアップロードし、ダウンロードURLのみFirestoreに保存する方式
- **メリット**: ドキュメントサイズが小さくなり、一覧取得が軽量化。画像のCDN配信によるパフォーマンス向上
- **デメリット**: Firebase Blazeプラン（従量課金、クレジットカード登録必須）へのアップグレードが必要
- **見送り理由**: 個人利用・少数カードの範囲ではdata URLで実用上問題なし。カード数増加やパフォーマンス劣化が顕著になった場合に移行を検討する
